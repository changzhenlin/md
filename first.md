# skil.md - Vibe Coding 项目进度跟踪标准化指南

**版本**：1.0  
**最后更新**：2026-03-30  
**适用范围**：适用于任何 vibe coding 编辑器（Cursor、Windsurf、Continue、Grok 等），无论切换何种工具或中断任务，均可通过本指南及项目专属进度文档无缝恢复工作。

## 1. 核心目的
本文件是所有项目启动时的**强制前置阅读文档**。  
任何编辑器在开始新项目或恢复现有项目时，必须**先完整阅读本 skil.md**，再执行后续操作。  
通过标准化生成和维护一个项目专属的 Markdown 进度文档，实现：
- 任务中断后（关闭编辑器、切换设备、切换工具）仍能清晰掌握当前进度；
- 所有步骤、状态、决策记录均以纯文本形式持久化，无需依赖任何编辑器的记忆或历史记录。

## 2. 标准工作流程（必须严格遵守）

### 步骤 1：项目启动 / 恢复时（每次会话开始）
1. **强制阅读**：先完整阅读当前目录下的 `skil.md`（若不存在则从项目模板中复制）。
2. **确定/创建进度文档**：
   - 项目专属进度文档统一命名为：`[项目名称]_PROGRESS.md`（英文大写，示例：`MyApp_PROGRES.md` 或 `客户CRM系统_PROGRESS.md`）。
   - 文件必须放置在**项目根目录**。
   - 若文件已存在，则直接读取最新状态；若不存在，则立即生成。

### 步骤 2：生成/更新进度文档
进度文档采用固定结构（见第 3 节模板）。  
编辑器必须：
- 将本次项目要完成的**所有步骤**完整列出（可分阶段、主任务、子任务）。
- 每个任务节点包含：
  - 编号（支持层级，如 1.1、1.2）
  - 清晰描述
  - 当前状态（`待办` / `进行中` / `已完成` / `阻塞` / `已取消`）
  - 预计完成时间（可选）
  - 实际完成时间 + 简要备注（完成时必填）
  - 依赖关系（可选）

### 步骤 3：任务执行与实时更新
- **每完成一个节点**（无论主任务或子任务），**立即**更新对应进度文档：
  - 将状态改为 `已完成`；
  - 记录完成时间和必要备注；
  - 若有后续行动或新发现的步骤，立即追加到文档末尾。
- **每完成一个主要阶段**，在文档顶部添加一句总结（例如：“阶段一已全部完成，当前进入阶段二”）。
- **遇到阻塞**时，明确记录阻塞原因、已尝试的解决方案及需要外部输入的内容。

### 步骤 4：任务中断或切换编辑器时
- 打开项目根目录下的 `[项目名称]_PROGRESS.md`；
- 从第一个状态为 `待办` 或 `进行中` 的节点开始继续工作；
- 无需任何额外说明，编辑器即可立即接续。

## 3. 进度文档标准模板（编辑器必须按此格式生成）

```markdown

**项目启动日期**：YYYY-MM-DD  
**当前编辑器**：（自动填写或手动记录）  
**上次更新**：YYYY-MM-DD HH:MM

## 项目概述
（简要描述项目目标、范围、技术栈、交付物）

## 总体进度
- 完成率：XX%  
- 当前阶段：第 N 阶段 / 共 M 阶段

## 详细任务清单

### 阶段 1：xxxx
1. [任务描述]  
   **状态**：已完成 / 待办 / 进行中  
   **完成时间**：YYYY-MM-DD HH:MM  
   **备注**：……

1.1 [子任务]  
   **状态**：已完成  
   **完成时间**：……

### 阶段 2：xxxx
（同上格式）

## 已完成里程碑
- [日期]：完成 xxx

## 待办 / 风险 / 待确认事项
（实时维护）

## 决策记录（重要）
- [日期]：决定采用 xxx 技术，原因：……

请先阅读项目根目录下的 `skil.md`，再继续本文件中的下一个待办节点。

若用户明确要求生成新步骤，编辑器需先将其追加到进度文档，再执行。
本 skil.md 文件本身不得被修改（除非版本升级），所有项目个性化内容均写入各自的 _PROGRESS.md。
